并发编程中的自旋锁、TicketLock、CLHLock、MCSLock

并发编程中的自旋锁、TicketLock、CLHLock、MCSLock

1. 自旋锁

自旋锁：是指一个线程在获取锁的时候，如果锁已经被其他线程获取，那么该线程将循环等待，然后不断的判断锁是否能够被成功获取，直到获取锁才会退出循环。

1.1 为什么需要自旋锁

阻塞或者唤醒一个 Java 线程需要操作系统切换 CPU 状态来完成，这种状态转换需要耗费处理器时间，如果同步代码块的内容过于简单，状态转换消耗的时间有可能比用户代码执行的时间还要长。

如果物理机器有多个处理器，能够让两个或两个以上的线程同时并行执行，我们可以让请求锁的线程不放弃 CPU 执行时间，而是一直尝试获取锁。

参照美团的自旋锁过程，画了一张图（好的博客从好的流程图开始，好的流程图从好的软件开始）

自旋锁本身是有缺点的，它并不能代替阻塞。虽然自旋避免了线程的开销，但是它需要占用处理器的时间。如果锁被占用的时间很短，自旋效果自然很好；反之，如果锁被占用时间很长，那么自旋线程只会白白浪费处理器资源。因此，自旋等待时间需要有一定限度。如果自旋超过了限定次数，没有成功获取锁，就应当挂起该自旋线程。

1.2 自适应自旋

在 JDK 6 中，默认开启了自旋锁，并且，自旋次数不再使用固定次数（之前默认为 10），而是动态调整的次数，所以被称为自适应自旋，因此不需要 -XX:PreBlockSpin 这个参数了。

自适应自旋的次数是由最近（注意，并不是上一次）在同一个锁上的旋转尝试的成功/失败率及所拥有者的状态来决定的。如果在同一个锁对象上，自旋等待刚刚成功获取过锁，并且持有锁的线程正在运行中，那么虚拟机就会认为这次自旋也很有可能再次成功，进而它将允许自旋次数更多。另一方面，如果对于某个锁，自旋很少成功获取过锁，那么以后要获取这个锁时将有可能直接省略掉自旋过程。

这里国内很多论坛都引用了《深入理解 Java 虚拟机》原话，原话是这样子的：

是由上一次在同一个锁上的自旋时间及锁的拥有者的状态来决定

但这一句话很明显是错误的，因为和后面说的互相矛盾：

另一方面，如果对于某个锁，自旋很少成功获取过锁，那么以后要获取这个锁时将有可能直接省略掉自旋过程。

这里说的“自旋很少成功”，肯定不单单是上一次。

这里我参考的是 Java 官网的介绍：

Adaptive spinning is an optimization technique where a two-phase spin-then-block strategy is used by threads attempting a contended synchronized enter operation. This technique enables threads to avoid undesirable effects that impact performance such as context switching and repopulation of Translation Lookaside Buffers (TLBs). It is “adaptive” because the duration of the spin is determined by policy decisions based on factors such as the rate of success and/or failure of recent spin attempts on the same monitor and the state of the current lock owner.

2. TicketLock

2.1 工作原理

排号自旋锁是基于先进先出的队列机制。工作原理为：有两个从 0 开始的整数。第一个整数是入队（queue）票证，第二个整数是出队（dequeue）票证。入队票证是线程在队列中的位置，出队票证是现在具有锁的票证或者队列位置。

当线程到达时，它以原子方式获取并递增入队票证（需要原子方式是因为预防两个线程获取同样的票证）。然后，它将增量之前的票证值和出队票证值进行比较，如果相等， 则允许线程进入临界区资源。如果它们不同，那么临界区中必然已经有另外一个线程，并且该线程正处于忙等待（busy-wait）或者谦让（yield）。当一个线程离开临界区资源时，它会自动增加出队票证。这允许下一个线程进入临界区。

2.2 代码实现

在 Java 中，我们借用 AtomicInteger 来实现票据。临界区资源我们使用 volatile int来实现。思路是将临界区资源递减，查看最后结果是否为 0；

public class TicketLock extends Thread {

    private AtomicInteger queueTicket = new AtomicInteger(0);
    private AtomicInteger dequeueTicket = new AtomicInteger(0);

    /**
     * 假设 i 为我们的临界区资源
     */
    public static volatile int i;

    /**
     * 判断线程池是否完成
     */
    private CountDownLatch countDownLatch;

    public TicketLock(int threadNumber) {
        i = threadNumber;
        countDownLatch = new CountDownLatch(threadNumber);
    }

    private void lock() {
        int currentTicket = queueTicket.getAndIncrement();
        while (currentTicket != dequeueTicket.get()) {

        }
    }

    private void unlock() {
        dequeueTicket.incrementAndGet();
    }

    @Override
    public void run() {
        lock();
        i--;
        // 模拟访问临界区需要花费较长时间
        try {
            Thread.sleep(1000L);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        unlock();
        System.out.println("线程解锁成功 ...");
        countDownLatch.countDown();
    }

    public static void main(String[] args) throws InterruptedException {
        // 请大家尝试将线程数量改成给10、100，结果 i 的值都为 0
        // 机器不好的同学不要轻易尝试 XD
        int threadNumber = 5;
        TicketLock lock = new TicketLock(threadNumber);
        ExecutorService executorService = Executors.newCachedThreadPool();
        for (int j = 0; j < threadNumber; j++) {
            executorService.submit(lock);
        }
        executorService.shutdown();
        lock.getCountDownLatch().await();
        // 打印结果，我们发现为 i 的值为 0
        System.out.println("i 的值为：" + i);
    }
}

2.3 思考入队和出队票据是否都需要原子性操作

首先，入队票据的读写肯定是需要原子性操作的，否则线程 A、B都可能读取到相同的入队票据，以至于造成线程不安全。

问题在于，出队票据是否需要进行原子性操作。我们不妨认为出队票据的修改，是基于自旋成功之后的，也就是说，只有自旋成功之后的线程，才有机会修改出队票据。因此，出队票据并不需要原子性操作，我们可以将出队票据的值操作改为非原子性的：

public class TicketLock extends Thread {

    private AtomicInteger queueTicket = new AtomicInteger(0);

    /**
     * 注意这里的 volatile
     */
    private volatile int dequeueTicket = 0;

    /**
     * 假设 i 为我们的临界区资源
     */
    public static volatile int i;

    /**
     * 判断线程池是否完成
     */
    private CountDownLatch countDownLatch;

    public TicketLock(int threadNumber) {
        i = threadNumber;
        countDownLatch = new CountDownLatch(threadNumber);
    }

    private void lock() {
        int currentTicket = queueTicket.getAndIncrement();
        while (currentTicket != dequeueTicket) {

        }
    }

    private void unlock() {
        dequeueTicket++;
    }

    @Override
    public void run() {
        lock();
        i--;
        try {
            Thread.sleep(100L);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        unlock();
        countDownLatch.countDown();
    }

    public static void main(String[] args) throws InterruptedException {
        int threadNumber = 50;
        TicketLock lock = new TicketLock(threadNumber);
        ExecutorService executorService = Executors.newCachedThreadPool();
        for (int j = 0; j < threadNumber; j++) {
            executorService.submit(lock);
        }
        executorService.shutdown();
        lock.getCountDownLatch().await();
        // 打印结果，我们发现为 i 的值为 0
        System.out.println("i 的值为：" + i);
    }

}

3. CLHLock

CLH 锁基于以下特点：

  • CLH 是一种自旋锁，能确保无饥饿，公平的先来先服务队列
  • 线程在局部变量中自旋，不断轮询前置节点的状态，如果发现前置节点已经释放了锁，则结束自旋获取锁
  • AQS 源码中使用的是 CLH 锁的变种

3.1 数据结构

CLH 锁内部有两个 ThreadLocal 变量，一个指向自己的节点（myNode），一个指向前置节点（myPred），一个原子引用变量（tail）指向队尾。CLH 内部还有一个 QNode 内部类，该内部类只有一个 volatile boolean lock。 

public class CLHLock {
    private final AtomicReference<QNode> tail;

    private final ThreadLocal<QNode> myPred;
    private final ThreadLocal<QNode> myNode;

    public CLHLock() {
        tail = new AtomicReference<>(new QNode());
        myNode = ThreadLocal.withInitial(QNode::new);
        myPred = ThreadLocal.withInitial(() -> null);
    }

    private static class QNode{
        private volatile boolean lock; 
    }

    // ...
}

3.2 CLHLock 获取锁过程

  1. 当某线程进入时，为该线程分配一个类型为 QNode 的节点（myNode）以及 myPred（初始值为空）
  2. 设置 lock 为 true，表示尝试获取锁
  3. 将当前线程的前置节点（myPred）指向 tail（上一个线程）；将 tail 指向该线程（myNode），表示该线程为当前队列的队尾（该过程为原子操作）
  4. 判断前置节点（myPred）的 lock 是否为 false。如果为 false，表明上个线程已经解锁，该线程可以加锁了；如果为 true，表明上个线程还未解锁，该线程将持续自旋。
public void lock() {
    QNode qNode = myNode.get();
    qNode.setLock(true);
    QNode predNode = tail.getAndSet(qNode);
    myPred.set(predNode);
    while (predNode.isLock());
}

3.3 CLHLock 释放锁过程

  1. 将当前线程的 lock 设置为 false
  2. 将自身节点指向前置节点
public void unlock(){
    QNode qnode = myNode.get();
    qnode.setLock(false);
    myNode.set(myPred.get());
}

3.4 完整代码

public class CLHLock {

    private final AtomicReference<QNode> tail;

    private final ThreadLocal<QNode> myPred;
    private final ThreadLocal<QNode> myNode;

    public CLHLock() {
        tail = new AtomicReference<>(new QNode());
        myNode = ThreadLocal.withInitial(QNode::new);
        myPred = ThreadLocal.withInitial(() -> null);
    }

    public void lock() {
        QNode qNode = myNode.get();
        qNode.setLock(true);
        // 获取并记录上一个节点的位置，并将 tail 指向当前节点（线程）
        QNode predNode = tail.getAndSet(qNode);
        // 设置上一个节点
        myPred.set(predNode);
        // 判断上一个节点的 lock 是否为 false，如果为 true，则该线程一直自旋等待
        while (predNode.isLock()) ;
    }

    public void unlock() {
        QNode qNode = myNode.get();
        qNode.setLock(false);
        // 防止死锁
        myNode.set(myPred.get());
    }

    private static class QNode {
        private volatile boolean lock = false;
        public boolean isLock() {
            return lock;
        }
        public QNode setLock(boolean lock) {
            this.lock = lock;
            return this;
        }
    }

}

3.6 一步一步看 unlock 方法里面的防止死锁问题

假设有两个线程 A、B 正在进行 lock、unlock 操作，现在线程 B 正在进行自旋操作，判断 A 是否释放了临界区资源。

while(myPred.isLock());

假设线程 A 释放资源时没有最后一行的操作：

public void unlock(){
    QNode qNode = myNode.get();
    qNode.setLock(false);
    // myNode.set(myPred.get());
}

现在，假设线程 A 释放了资源，但是，由于线程 B 出现了一些问题，或者说，线程 A 又很快的进行了下一轮的 lock() 操作：

现在，两个线程相互等待对方的 lock，造成了死锁。

正确方式是：在 unlock 中加上了最后一行：

public void unlock(){
    QNode qNode = myNode.get();
    qNode.setLock(false);
    myNode.set(myPred.get());
}

这样，Thread B 中的 myPred 和 Thread A 中的 myNode 就不是同一个对象了。

4. MCSLock

4.1 数据结构

与 CLHLock 非常类似，MCSLock 有一个内部类 Node，内部类中存放 volatile boolean lock 以及 volatile Node next 信息，MCSLock 有两个字段，原子引用变量 tail 指向队尾，线程局部变量 node。

public class MCSLock {

    private AtomicReference<Node> tail = new AtomicReference<>();
    private ThreadLocal<Node> node = ThreadLocal.withInitial(Node::new);

    private static class Node {
        private volatile boolean lock = false;
        private volatile Node next = null;
    }
}

4.2 MCSLock 加锁过程

当某个线程试图获取锁时：

  1. 设置当前节点的 lock 为 true
  2. 记录上一个节点的位置（tail），并将 tail 指向当前节点（原子操作）
  3. 判断上个节点是否存在，如果不存在，当前线程可直接访问临界区；如果存在，设置上个节点的 next 为当前节点，并自旋判断自身 lock 是否为 false
public void lock(){
    Node current = node.get();
    current.lock = true;
    Node pred = tail.getAndSet(current);
    if(pred != null){
        pred.next = current;
        while(current.lock);
    }
}

4.3 MCSLock 解锁过程

当某个线程解锁时：

  1. 判断当前线程是否有下个节点（next != null）
  2. 如果存在下一个节点，将下一个节点的 lock 置为 false ，并将当前节点的 next 置空。
  3. 如果不存在下一个节点，尝试将 tail 的值设置为空；如果 tail 被其他线程修改，当前线程会自旋等待自身 next 是否为空。
public void unlock(){
    Node current = node.get();
    if(current.next == null){
        if(tail.compareAndSet(current,null)){
            return ;
        }
        while(current.next == null);
    }
    current.next.lock = false;
    current.next = null;
}

4.4 完整代码

public class MCSLock {

    private AtomicReference<Node> tail = new AtomicReference<>();
    private ThreadLocal<Node> node = ThreadLocal.withInitial(Node::new);

    public void lock() {
        Node current = node.get();
        current.lock = true;
        Node pred = tail.getAndSet(current);
        if (pred != null) {
            pred.next = current;
            while (current.lock) {

            }
        }
    }

    public void unlock() {
        Node current = node.get();
        if (current.next == null) {
            if (tail.compareAndSet(current, null)) {
                return;
            }
            while (current.next == null) {

            }
        }
        current.next.lock = false;
        current.next = null;
    }

    private static class Node {
        private volatile boolean lock = false;
        private volatile Node next = null;
    }

}

5. 总结

  1. 线程切换需要耗费处理器时间，如果访问临界区资源的逻辑比较简单，那么消耗处理器所用时间可能要比访问临界区所用时间还要多，为此引入了自旋。
  2. 自旋可以让当前线程不放弃 CPU 等待时间，一直等待获取锁。
  3. 如果一直没有自旋成功，会消耗大量的处理器资源，为了减少这种消耗，引入了自旋次数，JDK 6 之前默认自旋次数为 10。JDK 6 之后引入了自适应自旋锁。
  4. 自适应自旋次数是最近在同一个锁上的旋转成功/失败率以及所持有者的状态决定的。
  5. 常用的自旋实现有 TicketLock ，CLHLock，MCSLock
  6. TicketLock 实现原理是入队票据和出队票据，每当线程进入时，会记录当前入队票据并将入队票据加一，并将之前的入队票据与出对票据做对比，如果相等，则自旋成功。线程退出时，会将出队票据递增。
  7. CLHLock 实现原理是维护一个 FIFO 的队列，当前线程进入时，会将自身 lock 设置为 true，并等待上一个线程的 lock 为 false；线程退出时，将自身 lock 设置为 false，然后将自身节点指向前一个结点。
  8. MCSLock 原理与 CLHLock 类似，但是 MCSLock 会自旋自身 lock；线程退出时，如果有下一个节点，将下一个节点的 lock 设置为 false，并将自身的 next 置空。如果没有下一个节点，会尝试将 tail 置空。
